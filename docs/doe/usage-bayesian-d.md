## 1. What is Bayesian D?

Standard D-opt maximizes `det(XᵀX)`. When prior knowledge suggests quadratic terms or
interactions may be negligible, **Bayesian D** `max det(XᵀX + K)` incorporates prior
precision K—a p × p diagonal matrix where each diagonal element represents the
**prior precision** (= 1/prior variance) for the corresponding parameter.

Typical DuMouchel-Jones convention:

| Term type | K_jj |
|---|---|
| intercept    | 0   |
| main effect  | 0   |
| 2-factor interaction | τ² |
| quadratic term | τ² |
| nested       | τ² |

τ² ≈ 1.0 (coded space assumption) is the standard starting value. `priorPrecisionDefault`
constructs K under this convention; for term-specific precision, pass a term-to-precision
function to `priorPrecisionFromTerms`. K = 0 degenerates to classic D
(`bayesianDValueM (PriorPrecision 0) x == det(X'X)`, backward-compatibility verified).

## 2. Compound (Alphabetic) Criterion

`Compound [(weight, criterion)]` combines multiple criteria into a weighted sum. Weights
are assumed positive; negative or non-unit-sum weights are normalized by
`Compare.normalizeCompoundWeights` (negatives clipped to 0 and re-normalized).

**Caution**: inner criterion scale is user's responsibility to align
(e.g., D is det; A is 1/trace—different dimensional units). Normalizing to efficiency
form ([0, 1] range) before weighting produces a meaningful weighted sum.

## 3. Multivariate Process Capability (Cp) Integration

Phase 23-d's `Design.Quality.processCapabilityMultivariate` (Mahalanobis-based
MCp / MCpk) is a **post-hoc evaluation** after applying observed y (multivariate response)
to designs generated by Custom Design. Automatic `Compare` integration is out of scope
(Cp requires y observation; Compare accepts design x only; requires signature extension
like `compareDesignsWithResponses`—pending canvas coordination).

## 4. Known Limitations

- K assumes diagonal (non-diagonal accepted but interpretation requires care)
- Bayesian D's D-efficiency (Compare `dcEffTable` D column) uses **classic D**.
  BayesianD-to-BayesianD comparison requires direct `crCriterionValue` inspection
- Compound weight normalization is linear sum. Geometric-mean based (log-composition)
  deferred to future
- Multivariate Cp Compare integration deferred (requires API extension)

## 5. DuMouchel-Jones §2.2 Convention (Phase 28-12)

DJ (1994) §2.2 requires prior τ² interpretable as "effect size 1σ_error," applying
these transformations to potential terms (`TInter` len≥2 / `TPower` k≥2 / `TNested`):

1. **centering**: subtract mean over candidate set
2. **orthogonalization vs. primary**: LS orthogonalize against primary columns
   (TIntercept / TMain / TInter len 1)
3. **range = 1 normalization**: divide by (max − min) after orthogonalization

Paper §2.2 example (primary {1, x}, candidates {-1, -0.5, 0, 0.5, 1}):
- `TPower x 2` → `z₁ = x² − 0.5`
- `TPower x 3` → `z₂ = (x³ − 0.85x)/0.6`

hanalyze's `TPower` / `TInter` return **raw quantities** (e.g., raw x², raw xy).
Applying the same K directly computes a different quantity than DJ intends. To match
literature values, use `djTransformColumns` (fit from candidates, apply to design X).
`djFitTransform` / `djApplyTransform` separate fit and apply for coefficient reuse
(useful comparing multiple designs on same candidate set).

**Auto-apply (Phase 28-12 complete)**: Pass `cdsDJConvention = True` in `CustomDesignSpec`.
`coordinateExchange` auto-fits DJ transforms from the candidate set, applies
`djApplyTransform` post-expansion in criterion evaluation. `cdMatrix` remains raw;
`crCriterionValue` is −det after DJ transformation. Manual invocation needed only
for **post-hoc evaluation of existing designs** (e.g., literature design comparison).

**Bench evidence** (bench/custom-design/REPORT.md §27-3): After auto-applying DJ,
hanalyze design vs. DuMouchel-Jones (1994) Example 3 "Both" design:
det(X_t' X_t + K) = **exact match (ratio = 1.0000)**—hanalyze discovers design of
equivalent quality under DJ's intended criterion.
